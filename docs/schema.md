# Task Manager Technical Design

## 1. Data models

### Task
| Property | Type | Required | Validation rules |
|---|---|---|---|
| id | string | Yes | Must be a non-empty string matching `/^[1-9][0-9]*$/` (numeric string form). Generated from `nextId`; required for update/delete lookups; immutable after creation. |
| title | string | Yes | For create: required and must be a string with `trim().length > 0`. For update: optional, but if present must still satisfy non-empty trimmed rule. |
| description | string | Yes | Optional input. If omitted on create, default to `''`. If provided, must be a string (empty string allowed after trimming). |
| status | 'todo' \| 'in-progress' \| 'done' | Yes | Optional input on create, defaults to `todo`. If provided on create/update, must be exactly one of: `todo`, `in-progress`, `done`. |
| priority | 'low' \| 'medium' \| 'high' | Yes | Optional input on create, defaults to `medium`. If provided on create/update, must be exactly one of: `low`, `medium`, `high`. |
| createdAt | string (ISO 8601 timestamp) | Yes | Must be a valid ISO 8601 UTC timestamp string generated by the system at create time. Never accepted from user input; immutable on update. |
| updatedAt | string (ISO 8601 timestamp) | Yes | Must be a valid ISO 8601 UTC timestamp string generated by the system. Set on create and replaced on each successful update; never accepted from user input. |

### In-memory store
| Property | Type | Required | Validation rules |
|---|---|---|---|
| tasks | Task[] | Yes | Contains all tasks in insertion order unless list sorting is applied. |
| nextId | number | Yes | Positive integer incremented after each successful create. |

## 2. File structure

```text
src/
  index.js                  # CLI entrypoint; parses args, invokes services, writes stdout/stderr, sets exit code
  cli/
    parseArgs.js            # Converts raw argv into command + normalized options payload
    helpText.js             # Exports usage/help content for commands and options
  domain/
    taskModel.js            # Defines task creation/update shaping and default field values
    validators.js           # Field, filter, sort, and command validation logic
    priorityOrder.js        # Priority rank constants/comparison helpers (high > medium > low)
  services/
    taskService.js          # Core use cases: create, list (filter/sort), update, delete
  store/
    memoryStore.js          # In-memory state container (`tasks`, `nextId`) and basic store accessors
  format/
    taskFormatter.js        # Formats task output and empty-state text for CLI display
  utils/
    dateTime.js             # Timestamp helper(s) for ISO generation
tests/
  validators.test.js        # Unit tests for domain validation rules
  taskService.test.js       # Unit tests for CRUD, filtering, sorting, and not-found paths
  parseArgs.test.js         # Command parsing smoke tests
```

## 3. Module responsibilities

### `src/index.js`
- Exports: none (runtime entrypoint).
- Responsibilities: read process arguments, call `parseArgs`, dispatch to `taskService`, format output, map errors to stderr + exit code.
- Depends on: `src/cli/parseArgs.js`, `src/cli/helpText.js`, `src/services/taskService.js`, `src/format/taskFormatter.js`.

### `src/cli/parseArgs.js`
- Exports: `parseArgs(argv)`.
- Responsibilities: parse command/options, reject unknown flags, normalize option names/values for service input.
- Depends on: `src/domain/validators.js` (command/argument validation).

### `src/cli/helpText.js`
- Exports: `getHelpText()` (or constants for usage text).
- Responsibilities: single source of truth for CLI usage instructions.
- Depends on: none.

### `src/domain/taskModel.js`
- Exports: `buildNewTask(input, now, id)`, `buildTaskPatch(input, now)`.
- Responsibilities: apply defaults (`status`, `priority`, `description`), enforce allowed writable fields, ensure immutable fields remain immutable.
- Depends on: `src/domain/validators.js`.

### `src/domain/validators.js`
- Exports: validators for title, status, priority, id, filters, sort options, command names.
- Responsibilities: validate and normalize incoming data; throw structured validation errors.
- Depends on: `src/domain/priorityOrder.js` (for priority-related checks where needed).

### `src/domain/priorityOrder.js`
- Exports: `PRIORITY_RANK`, `comparePriority(a, b)`.
- Responsibilities: define canonical priority sorting behavior.
- Depends on: none.

### `src/services/taskService.js`
- Exports: `createTask`, `listTasks`, `updateTask`, `deleteTask`.
- Responsibilities: orchestrate validation, persistence calls, filtering, sorting, and domain-level errors.
- Depends on: `src/store/memoryStore.js`, `src/domain/taskModel.js`, `src/domain/validators.js`, `src/domain/priorityOrder.js`, `src/utils/dateTime.js`.

### `src/store/memoryStore.js`
- Exports: store access/manipulation helpers (for example `getState`, `addTask`, `findTaskById`, `removeTaskById`, `updateTaskById`).
- Responsibilities: maintain in-memory `tasks` and `nextId`.
- Depends on: none.

### `src/format/taskFormatter.js`
- Exports: formatters for task list, single task output, empty-state message.
- Responsibilities: CLI-safe, readable string formatting only (no business logic).
- Depends on: none.

### `src/utils/dateTime.js`
- Exports: `nowIso()`.
- Responsibilities: return current timestamp in ISO 8601 format.
- Depends on: none.

### `tests/*.test.js`
- Exports: none.
- Responsibilities: validate one behavior per test function using `node:test` and `assert`.
- Depends on: modules under `src/`.

## 4. Error handling strategy

### Error shape
All domain/service errors use `Error` objects augmented with:
- `code`: machine-readable code
- `message`: user-facing summary
- `details` (optional): structured context for diagnostics

### Error types and where they are thrown
1. `INVALID_COMMAND`
- Thrown in `src/cli/parseArgs.js` when command is not one of `create`, `list`, `update`, `delete`.

2. `INVALID_ARGUMENT`
- Thrown in `src/cli/parseArgs.js` when unknown flags or invalid command option structure is detected.

3. `VALIDATION_ERROR`
- Thrown in `src/domain/validators.js` for invalid field values (for example invalid `status`, `priority`, empty `title`, invalid filters/sort values).
- May also be thrown from `src/domain/taskModel.js` when blocked fields (`createdAt`, `updatedAt`) are provided by input.

4. `TASK_NOT_FOUND`
- Thrown in `src/services/taskService.js` when update/delete targets a missing `id`.

5. `INTERNAL_ERROR`
- Created in `src/index.js` as a fallback wrapper for unexpected exceptions not already mapped to known codes.

### Handling flow
1. `src/index.js` parses args and executes the selected service action.
2. Known errors (`INVALID_COMMAND`, `INVALID_ARGUMENT`, `VALIDATION_ERROR`, `TASK_NOT_FOUND`) are printed to stderr with exit code `1`.
3. Unexpected errors are logged with `console.error`, mapped to `INTERNAL_ERROR`, and returned with a generic message and exit code `1`.
4. Successful commands write to stdout and exit with code `0`.
