What Is GitHub Copilot?

AI pair programmer

GitHub Copilot uses large language models trained on public code to suggest completions and answer questions directly inside your editor.

Powered by Microsoft & OpenAI

Copilot is powered by OpenAI Codex and GPT-4 class models, delivered via Microsoft Azure infrastructure, and integrated natively into GitHub.

Code completions

Ghost-text suggestions that appear as you type, completing lines, functions, and entire files.

Copilot Chat

A conversational interface for code explanation, refactoring, test generation, and more.

Key insight: The quality of Copilot's output depends heavily on the context you provide. This workshop teaches you how to provide better context.

01

Prompt Engineering → Planner Agent

Learn prompt strategies and Copilot modes while building a Planner Agent for the Planning phase.

👉 Open your GitHub Issue to start this lab

SDLC Phase: Planning • Duration: ~30 minutes

What Is a Prompt?

A prompt is any request you make to GitHub Copilot. There are two primary forms.

Inline completions

Ghost text that appears as you type in the editor. Copilot infers the prompt from the surrounding code, comments, and open files.

Copilot Chat

Natural language questions typed in the Chat panel, the inline editor (Ctrl+I), or on GitHub.com.

The strategies in this exercise apply primarily to Copilot Chat. For inline completions, the best context comes from well-named variables, comments, and open relevant files.

Source: Prompt engineering for GitHub Copilot Chat

Copilot Interaction Modes

GitHub Copilot supports five interaction modes, each suited for different tasks.

ModeWhereBest for
Inline suggestionsEditor (ghost text)Auto-completing code as you type
AskChat panelResearch and questions without code changes
AgentChat panelMulti-step tasks with file edits and terminal commands
PlanChat panelComplex task planning before implementation
Inline chatEditor (Ctrl+I)Quick, scoped edits in the current file
Plan mode vs. custom Planner Agent: Plan mode is a built-in Copilot mode for breaking down tasks. A custom Planner Agent goes further: it carries your project context, enforces specific templates, and embeds domain knowledge. You build both in Exercise 1.

Six Prompt Engineering Strategies

#StrategyKey idea
1Start general, then get specificState the goal first; add constraints and requirements after
2Give examplesShow expected inputs and outputs directly in the prompt
3Break tasks into stepsCompose complex features from smaller, focused prompts
4Avoid ambiguityName functions, files, and types explicitly
5Indicate relevant codeOpen only relevant files; attach context with #file
6Experiment and iterateFollow up in the same conversation to refine results
These strategies apply across every SDLC phase: writing implementation code, generating tests, requesting documentation, or explaining code during review.

Strategy 1 in Practice: Specificity

Vague prompt 
Write a function to
validate input

Copilot makes assumptions about type, language, and rules. The result may not match your needs.

Specific prompt 
Write a JavaScript function
that validates an email address.

The function should:
- Accept a single string `email`
- Return true if format is valid
- Return false otherwise
- Use no external libraries
- Include a JSDoc comment

Strategy 2 in Practice: Give Examples

I have a list of user objects with this shape:
  { id: number, name: string, role: "admin" | "viewer" }

Write a function `filterAdmins(users)` that:
- Accepts an array of these objects
- Returns only users where role === "admin"

Example input:
  [{ id: 1, name: "Alice", role: "admin" },
   { id: 2, name: "Bob",   role: "viewer" }]

Expected output:
  [{ id: 1, name: "Alice", role: "admin" }]
Providing a concrete example removes guesswork and produces output that matches your data model immediately.
02

Custom Instructions → Architect Agent

Create repository-wide conventions and build an Architect Agent for the Design phase.

👉 Open your GitHub Issue to start this lab

SDLC Phase: Design • Duration: ~25 minutes

What Is copilot-instructions.md?

  • A plain Markdown file located at .github/copilot-instructions.md
  • Automatically included in every Copilot Chat request in the repository
  • Eliminates repetitive context you would otherwise add to every prompt
  • Supported in VS Code, Visual Studio, JetBrains IDEs, and GitHub.com
  • Also used by Copilot code review features
Scope: These instructions apply to Copilot Chat only. They do not apply to inline completion suggestions (ghost text).

What to Include

✓ Good to include

  • Language and runtime versions
  • Code style rules (indentation, quotes, semicolons)
  • Framework and library choices
  • Error handling patterns
  • Test runner and conventions
  • File and directory naming conventions
  • Output format preferences (e.g., always include JSDoc)

✗ Avoid

  • References to specific code paths (use path-specific files instead)
  • Conflicting or contradictory instructions
  • Very long files (included in every request, affecting context limits)
  • Secrets, tokens, or credentials

Example copilot-instructions.md

# Copilot Instructions

## Stack
- Language: TypeScript 5.x, Node.js 22
- Framework: Express 5
- Database: PostgreSQL via `pg` (no ORM)
- Test runner: Vitest

## Code Style
- Use 2-space indentation and single quotes.
- Always add JSDoc comments to exported functions.
- Prefer `async`/`await` over `.then()` chains.

## Error Handling
- Throw typed error classes, never plain strings.
- Log errors with the `logger` module, not `console.log`.

Verifying Your Instructions

1
Ask Copilot Chat a question that should trigger your instructions (e.g., "Based on the project conventions, describe the technology stack and data model for this project").
2
Click Used N references at the top of the Copilot response.
3
Confirm that .github/copilot-instructions.md appears in the reference list.
4
Verify that the response respects your instructions (correct language, style, patterns).
If the file does not appear, confirm the path is exactly .github/copilot-instructions.md. The path is case-sensitive.
03

Path-Specific Instructions → Developer Agent

Create per-directory coding rules and build a Developer Agent for the Implementation phase.

👉 Open your GitHub Issue to start this lab

SDLC Phase: Implementation • Duration: ~30 minutes

How Path-Specific Instructions Work

  • Files live in .github/instructions/
  • File names must end with .instructions.md
  • A YAML front matter block specifies the glob pattern via applyTo
  • When a matching file is open in the editor, both the repository-wide and path-specific instruction sets are active simultaneously
---
applyTo: "**/*.py"
---

# Python File Instructions

- Follow PEP 8 style conventions.
- Use 4-space indentation and type annotations on all arguments.
- Prefer `pathlib.Path` over the `os` module for file operations.
- Use `pytest` for all tests.

File Locations and Patterns

.github/
  copilot-instructions.md            ← repository-wide (all Chat requests)
  instructions/
    python.instructions.md           ← applyTo: "**/*.py"
    tests.instructions.md            ← applyTo: "**/*.test.{js,ts}"
    migrations.instructions.md       ← applyTo: "db/migrations/**"
    react-components.instructions.md ← applyTo: "src/components/**/*.tsx"
Multiple path-specific files can be active at the same time if multiple patterns match the currently open file.

Three Types of Keywords

SyntaxNamePurpose
@ Chat participant Scopes the prompt to a domain expert (workspace, GitHub, VS Code)
/ Slash command Triggers a specific, predefined action (explain, fix, test)
# Chat variable Attaches specific context to the prompt (file, selection, codebase)
These keywords reduce the amount of natural language explanation needed and give Copilot a clearer, more structured input.

Chat Participants (@)

@workspace

Searches and understands all files in your current VS Code workspace.

@workspace Where is the
user authentication logic?
@workspace How is error
handling implemented?

@github

Accesses GitHub-specific features: issues, PRs, code search, and Bing web search.

@github What open issues
mention authentication?
@github What skills are
available in Copilot?

Type @ in the Chat prompt box to see all available participants for your editor.

Slash Commands (/)

CommandActionSDLC Phase
/explainExplains the selected code or file in plain languageReview Maintain
/fixAnalyzes and suggests a fix for a bug in the selected codeImplement Test
/testsGenerates unit tests for the selected codeTest
/docAdds documentation comments (JSDoc, docstrings) to the selectionDocument
/newScaffolds a new file, project, or component structurePlan & Design
/clearStarts a new chat session, clearing previous contextany phase

Type / in the Chat prompt box to see all available commands.

Chat Variables (#)

VariableWhat it attaches to the prompt
#fileA specific file from the workspace (e.g., #file:auth.ts)
#selectionThe currently selected code in the active editor
#codebaseSemantic search across all files in the workspace
#editorThe contents of the currently active editor tab
 
#file:src/models/task.js /tests Write unit tests for
the Task model using the built-in Node.js test runner.
Combining variables and slash commands in one prompt gives Copilot maximum context with minimum typing.

Test Generation & Execution Workflow

Copilot can generate tests, save them, and help you iterate on failures.

1

Generate
Use /tests or a prompt to generate a test file

2

Save
Save tests alongside the source with a .test suffix

3

Run
Execute tests in the terminal and review results

4

Iterate
Paste failures into Copilot, fix, and re-run

# Step 1: Generate tests
#file:src/models/task.js /tests

# Step 3: Run tests
node --test tests/task.test.js

# Step 4: Fix failures
These tests failed: [paste error]. Explain and fix.
You practice this workflow in Exercises 1, 4, and 5.

Reusable Prompt Files (.prompt.md)

Save complex prompts as files and invoke them with / in Copilot Chat.

How they work

  • Files live in .github/prompts/ with .prompt.md suffix
  • YAML front matter defines description, agent, and argument-hint
  • Markdown body contains the reusable prompt instructions
  • Reference workspace files with Markdown links for extra context
---
description: Generate unit tests
agent: agent
argument-hint: File path to test
---

# Generate Unit Tests

1. Read the target file.
2. Identify all exported functions.
3. Generate tests that cover:
   - Normal inputs
   - Edge cases
   - Error conditions
4. Use the project's test runner.
Instructions vs Prompts: Instructions define how tasks should be done (conventions). Prompts define what task to perform (reusable workflows).

Custom Agent Modes (.agent.md)

Create specialized Copilot personalities with specific tools, response formats, and domain expertise.

How they work

  • Files live in .github/agents/ with .agent.md suffix
  • YAML front matter defines name, description, and tools
  • Body defines the agent's personality, rules, and response format
  • Select from the agent dropdown in Copilot Chat
---
name: code-reviewer
description: Reviews code quality
tools: ["search"]
---

# Code Review Agent

## Response Format
**SUMMARY:** One-sentence overview.
**ISSUES:** Numbered list (severity).
**SUGGESTIONS:** Fixes with examples.
**VERDICT:** APPROVE or CHANGES.

  • AI-powered review that runs alongside human reviewers on pull requests

Agent Orchestration → Full Lifecycle

Build an Orchestrator Agent and deliver a new feature using all SDLC agents together.

👉 Open your GitHub Issue to start this lab

What Are Custom Agents?

  • Specialized Copilot assistants with their own identity, tools, and behavior
  • Defined as .agent.md files with YAML frontmatter and a Markdown prompt
  • Stored in .github/agents/ for workspace scope or in your user profile for global scope
  • Available in VS Code, JetBrains, Eclipse, Xcode, and on GitHub.com with the coding agent
  • Create agents for testing, debugging, planning, code review, documentation, and more
Custom agents go beyond instructions: they create an entirely new Copilot personality with restricted (or expanded) tool access.

Anatomy of a .agent.md File

---
name: test-specialist
description: Focuses on test coverage and testing best practices
tools: ["read", "edit", "search", "runTests"]
---

You are a testing specialist. Your responsibilities:

- Analyze existing tests and identify coverage gaps
- Write unit tests, integration tests, and end-to-end tests
- Focus only on test files. Do not modify production code.
PropertyRequiredDescription
nameNoDisplay name (defaults to filename)
descriptionYesBrief explanation shown in the agent dropdown
toolsNoList of tools to restrict access (omit for all)
modelNoAI model to use (VS Code and JetBrains only)

Custom Agents vs Other Copilot Files

FilePurposeUsed by
.agent.mdDefine specialized agents with tools and behaviorsCopilot Chat + coding agent
copilot-instructions.mdRepository-wide context for all interactionsCopilot Chat (interactive)
.instructions.mdPath-specific rules for certain file typesCopilot Chat (matching files)
.prompt.mdReusable prompt templatesCopilot Chat (manual invocation)
AGENTS.mdRepository onboarding for autonomous agentsAI coding agents
All five file types can coexist in the same repository. Each serves a different purpose.

Where to Store Custom Agents

LocationScope
.github/agents/ in a repositoryAvailable in that repository (workspace)
VS Code user profile folderAvailable across all your workspaces (personal)
.github-private repositoryAvailable across all repos in the organization

Custom agents also work with the Copilot coding agent on GitHub.com. When assigning an issue to Copilot, select a specific agent from the dropdown to control how the coding agent approaches the task.

Explore 100+ community agents at github/awesome-copilot

Tips for Effective Agents

Do

  • Give the agent a clear role and identity
  • Restrict tools to what the task needs
  • Structure prompts with Markdown headings
  • Use descriptive, lowercase filenames
  • Test with representative prompts

Avoid

  • Vague identities ("Help with code")
  • Omitting the description property
  • Prompts over 30,000 characters
  • Granting all tools to focused agents
  • Skipping testing before sharing
A planning agent should not execute code. A documentation agent should not modify source files. Use tool restrictions to enforce boundaries.
